# parallel

Fan-out для независимых задач в рамках одного проекта: по субагенту на
задачу, каждый в своём git worktree (`isolation: "worktree"`), запуск одним
сообщением — агенты работают конкурентно и не мешают друг другу правками
файлов. Worktree создаются от репозитория, из которого вызван скилл, —
вызывать из корня проекта, над которым идёт работа.

## Формат вызова

```
/parallel задача 1; задача 2; задача 3
/parallel --model sonnet задача 1; задача 2
/parallel
```

- Задачи разделяются `;` или переводами строк.
- `--model sonnet|opus|haiku` — опционально, модель для всех агентов
  (по умолчанию агенты наследуют модель сессии).
- Без аргументов — спросить у пользователя список задач.

## Instructions

### Шаг 1: Разбор задач

- Распарсить аргументы в список задач (разделители: `;` или переводы строк).
- Если задача одна и монолитная — предложить декомпозицию на независимые
  части и согласовать её, либо уточнить, что параллелить.
- Задачи должны быть независимыми: не трогать одни и те же файлы по замыслу.
  Если явно пересекаются — предупредить, что мерж веток будет конфликтным.
- Если задач больше 6 — предупредить про расход квоты (N агентов ≈ N× токенов)
  и получить подтверждение перед запуском.

### Шаг 2: Pre-flight

- Убедиться, что текущая директория — git-репозиторий.
- Если в корне репозитория есть `.env*`-файлы, но нет `.worktreeinclude` —
  предупредить: агенты в worktree не получат env-файлы. Предложить создать
  `.worktreeinclude` с их перечислением (однократно, не блокировать запуск).
- Worktree по умолчанию ветвится от `origin/HEAD`. Если задачи должны
  опираться на текущую ветку, а в ней есть незапушенные коммиты
  (`git log @{push}.. --oneline` непуст или ветка без upstream) —
  предупредить и предложить на выбор: запушить ветку, либо включить в
  настройках проекта `{"worktree": {"baseRef": "head"}}` (ветвление от
  локального HEAD).

### Шаг 3: Запуск

Одним сообщением — по вызову инструмента Agent на каждую задачу:

- `subagent_type`: `general-purpose`, либо более специализированный агент
  проекта, если задача явно под него подходит (например `symfony-ddd-developer`).
- `isolation`: `"worktree"` — обязательно.
- `model`: значение из `--model`, иначе не указывать.
- `description`: 3–5 слов по сути задачи.

Шаблон промпта для каждого агента:

```
Ты работаешь в изолированном git worktree — правь файлы свободно,
параллельные агенты тебе не мешают.

Задача: {текст задачи}

Требования:
1. Выполни задачу полностью; прогони релевантные проверки (тесты, линтер),
   если они применимы к изменённому коду.
2. Закоммить ВСЕ изменения в worktree одним осмысленным коммитом —
   незакоммиченный результат легко потерять.
3. В финальном ответе укажи: ветку и путь worktree (`git worktree list`,
   `git branch --show-current`), что сделано, что проверено, известные риски.
```

Агенты выполняются в фоне — дождаться завершения всех, промежуточно не
опрашивать.

### Шаг 4: Отчёт

- Сводка по всем агентам: задача → ветка / путь worktree → краткий итог.
- Если агент упал или вернул пустой результат — сказать об этом явно.
- Напомнить следующий шаг: смержить ветки (или забрать изменения cherry-pick'ом),
  после интеграции удалить worktree — `/worktree-cleanup` или
  `git worktree remove <путь>`.
